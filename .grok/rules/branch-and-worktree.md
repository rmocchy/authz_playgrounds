# Branch & Worktree 運用

新規タスクを得たときは、次を**必ず**守る。実装・動作確認・検証はメインの作業ツリー上で行わない。

## 必須

1. **新しい branch を切る**  
   - タスク着手時に `main`（または指定された base）から新規 branch を作成する。  
   - 既存の無関係な branch や dirty な作業ツリー上で実装を始めない。  
   - ブランチ名は内容が分かるものにする（例: `feat/...`, `fix/...`, `chore/...`, `docs/...`）。

2. **worktree 上で実装・動作確認する**  
   - 実装・テスト・ビルド・動作確認は **git worktree** 上で行う。  
   - 手順の例:  
     1. 新規 branch を作成（または worktree 作成時に `-b` で同時作成）  
     2. `git worktree add <path> <branch>` で専用 worktree を用意  
     3. その worktree のパスを cwd にして編集・コマンド実行・検証する  
   - subagent を使う場合は `isolation: "worktree"` を優先する（可能なツールがあるとき）。  
   - リポジトリの root 作業ツリー（clone 本体）で直接実装・検証しない。

3. **人間の明示指示が最優先**  
   - ユーザーが「このツリーでやって」「branch 不要」「main で直して」などと**明示**した場合は、その指示に従う。  
   - 明示が無い限り、上記 1・2 を適用する。

## 適用範囲

- **適用する**: コード変更、設定変更、テスト追加、ドキュメントの実質的な更新、Design Doc の作成・更新、バグ調査に伴う修正 など、成果物を残す作業。  
- **適用しなくてよい（既定）**: 読み取り専用の調査・説明のみ、ルールやファイルの所在確認だけ、ユーザーが「今ここですぐ見て」と現在ツリーでの確認だけを求めている場合。  
- 判断に迷ったら worktree + 新規 branch を選ぶ。

## 作業終了時

- worktree を使い終わったら、マージや PR 方針が決まったあとで `git worktree remove` 等により片付ける（ユーザー方針があればそれに従う）。  
- 未コミットの変更を残したまま worktree を捨てない。

## 禁止・注意

- 無関係な branch にコミットを混ぜない。  
- force-push や共有 branch の書き換えは、ユーザーの明示なしに行わない。  
- 秘密情報を branch 名・worktree パス・コミットに含めない。
